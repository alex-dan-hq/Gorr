Operating Model / One Store Day

A store execution layer that turns fragmented shift reality into structured handover, prioritised work, live execution visibility and evidence-backed management control.

Time is the backbone. Product layers sit on the day where they operate, so handover, labour, execution and management form one continuous system.

Product Operating Layers

Each column is a transformation layer: it takes messy operating reality, changes its shape, and hands a cleaner signal to the next part of the system.

From: fragmented night memory + morning rediscovery
To: structured night state + actionable morning handover

Wedge: Night -> Morning Handover

State A

Night execution exists as fragmented memory, verbal updates, paper notes, WhatsApp messages and disconnected system traces. Morning team reconstructs the store state through walks, questions and blame.

State B

Night execution becomes a structured, evidence-backed operational state that the morning team can understand, acknowledge and convert into first actions before trade risk compounds.

Steps 1-3 / establish state
Steps 4+ / transfer and close

1. Receive Incoming Night Context

Tasks:

  • understand what is carried over from day/evening shift;
  • receive delivery expectations;
  • check staffing: planned vs actual, no-shows, agency;
  • understand promo, compliance and Area Manager actions;
  • identify known constraints before the night starts.

Need:

The night shift should start from a clear plan of record, not from fragmented memory.

4. Generate Morning Handover

Tasks:

  • summarize top risks;
  • list unfinished work;
  • explain blockers and root causes;
  • attach evidence links;
  • preserve overnight decisions and reasoning;
  • suggest next actions;
  • route items to Store Manager, Morning Shift Leader, Fresh DM, Ambient DM, Dotcom or Deputy.

Need:

Handover must be receiver-centric: what the morning team needs to know, decide and do.

2. Capture Night Operational Reality

Tasks:

  • record deliveries: cages, shorts, damages, arrival time;
  • track fill progress by aisle / department;
  • capture blockers: late delivery, missing labels, wrong stock, broken equipment;
  • capture cold-chain / temperature checks;
  • record promo changeover status;
  • capture date-code / waste actions;
  • record backroom and overstock locations;
  • capture safety / security incidents.

Need:

What the night team physically knows must become structured operational state.

5. Confirm Acknowledgement / Contest Items

Tasks:

  • morning manager reads the handover;
  • accept or contest key items;
  • assign ownership for unresolved work;
  • move accepted items into morning priorities;
  • preserve disputed items for follow-up.

Need:

Close the feedback loop so "night said it was done" does not become blame without evidence.

3. Run Final Readiness Check

Tasks:

  • check floor clear;
  • confirm cages and cardboard removed;
  • confirm face-up / standards;
  • assess readiness by department;
  • identify unresolved risks before opening;
  • mark what is ready, at risk, not ready, or unknown.

Need:

The morning team needs to know whether the store is ready for trade, where it is not ready, and why.

Core Needs Covered

  1. Preserve night state
  2. Know morning readiness
  3. Transfer unfinished work
  4. Preserve evidence and decision context
  5. Create ownership for first morning actions

From: raw morning signals + manager judgement
To: current store state + ranked morning priorities

Morning Control Layer

State A

Morning team has a handover, scattered reports, staffing reality and first floor observations, but still has to manually reconstruct what matters first.

State B

Morning team has a unified current store state and a ranked set of first-window priorities that can be assigned, tracked and escalated.

Steps 1-3 / establish state
Steps 4+ / prioritise and align

1. Ingest Morning Handover

Tasks:

  • receive structured night handover;
  • review top risks;
  • review unfinished work;
  • check evidence and unresolved blockers;
  • identify contested or unclear items.

Need:

Morning team starts with a usable operating picture, not a verbal memory dump.

4. Identify Unknowns And Risks

Tasks:

  • separate confirmed problems from unknowns;
  • mark missing evidence;
  • flag high-risk blockers;
  • identify what needs verification;
  • avoid treating "no signal" as green.

Need:

Unknown state must become visible, otherwise the store mistakes silence for control.

2. Reconcile Live Morning Signals

Tasks:

  • check actual staffing vs rota;
  • check compliance status;
  • check dotcom / pick-wave pressure;
  • check queue / checkout readiness;
  • include first floor-walk observations;
  • identify source conflicts.

Need:

Handover must be combined with live morning reality before decisions are made.

5. Generate Morning Priority Stack

Tasks:

  • rank 3-7 priorities;
  • define why each matters now;
  • connect each priority to impact: availability, compliance, labour, customer, dotcom, safety;
  • define urgency;
  • define escalation threshold.

Need:

Morning team needs to know what matters first, not just what happened.

3. Build Current Store State

Tasks:

  • create readiness by department / zone;
  • mark ready, at risk, not ready, unknown;
  • identify open exceptions;
  • identify labour gaps;
  • identify customer / availability / compliance risk.

Need:

The store needs one current operating picture before managers start assigning work.

6. Align Morning Roles

Tasks:

  • show SM what requires intervention;
  • show Deputy / Duty Manager what needs movement;
  • show Shift Leader what to assign first;
  • route department-specific items to Fresh, Ambient, Checkout, Dotcom;
  • prepare priorities for task dispatch.

Need:

The same morning state must become role-specific action.

Core Needs Covered

  1. Convert handover into current state
  2. Combine night context with live morning signals
  3. Make unknowns visible
  4. Rank first-window priorities
  5. Align SM / Deputy / Shift Leader / DMs around the same picture

From: priorities spoken aloud
To: work assigned, tracked and closed

Execution Loop

State A

Morning priorities exist, but work is still assigned verbally, chased manually, interrupted by live trading and verified through memory or walks.

State B

Priorities become trackable work with owners, deadlines, status, blockers, proof and carry-over, so execution state is visible during the shift.

Steps 1-3 / dispatch and track
Steps 4+ / resolve and feed back

1. Convert Priorities Into Tasks

Tasks:

  • split each priority into concrete work;
  • define owner;
  • define location;
  • define deadline;
  • define dependency;
  • define required proof;
  • define escalation rule.

Need:

Priorities only create value when they become executable work.

4. Surface Blockers Early

Tasks:

  • capture blocker reason;
  • capture location;
  • identify who can remove blocker;
  • route to Shift Leader / DM / Deputy / SM;
  • update remaining work.

Need:

A blocker should appear while it can still be fixed, not after the deadline has failed.

2. Dispatch Work To Owners

Tasks:

  • assign tasks to Shift Leader, Department Manager, Team Leader or colleague;
  • support role-based assignment where exact person may change;
  • account for staggered arrivals;
  • make task context visible without repeating the morning brief.

Need:

The right person needs to know what to do, where, by when and why.

5. Capture Proof At The Right Level

Tasks:

  • self-report for low-risk work;
  • scan for stock / gap / pick routines;
  • photo for promo / standards;
  • checklist for safety / cleaning;
  • system proof for temp / queue / EPOS;
  • manager verification for high-risk items.

Need:

Not all work needs the same proof, but important work needs evidence.

3. Capture Execution Status

Tasks:

  • accepted;
  • in progress;
  • partial;
  • done;
  • blocked;
  • paused;
  • expired;
  • reassigned;
  • cancelled;
  • escalated.

Need:

Managers need to know whether work is actually moving, not just whether it was assigned.

6. Feed Execution Back Into Priorities

Tasks:

  • update current store state;
  • re-rank priorities if work is blocked or expired;
  • create new tasks if conditions change;
  • record checkout pull or interruption;
  • create carry-over if work cannot close.

Need:

Execution must be a live loop, not an end-of-shift report.

Core Needs Covered

  1. Convert priorities into executable work
  2. Make task ownership visible
  3. Track real execution state
  4. Surface blockers early
  5. Create proportionate proof
  6. Preserve unfinished work as carry-over

From: priorities without feasibility
To: executable trade-offs with clear authority

Labour / Authority / Escalation Layer

State A

Work is prioritised, but managers still rely on judgement, walking the floor and negotiation to know whether there are enough people, skills and authority to execute it.

State B

Priorities are checked against actual labour, skills, breaks, checkout pull, absence and decision rights, so trade-offs and escalations are explicit.

Steps 1-3 / establish capacity
Steps 4+ / decide and record

1. Check Actual Labour vs Plan

Tasks:

  • compare rota vs actual attendance;
  • identify no-shows, sickness, agency, late arrivals;
  • identify who is physically available;
  • identify gaps by department.

Need:

The store needs to know real capacity, not planned capacity.

4. Clarify Authority Boundaries

Tasks:

  • identify what Shift Leader can decide;
  • identify what Deputy can decide;
  • identify what requires Store Manager approval;
  • identify what must go to AM / HR / LP / compliance;
  • route escalation with evidence.

Need:

The person who owns the live problem often does not own the authority to close it.

2. Map Skills And Coverage

Tasks:

  • identify checkout-trained colleagues;
  • identify fresh / bakery / dotcom / keyholder / PLH / first-aider coverage;
  • identify break pressure;
  • identify single points of failure.

Need:

Labour is not interchangeable; the product must know who can actually do what.

5. Record Trade-Offs

Tasks:

  • log why one priority was delayed;
  • log labour redeployment reason;
  • log accepted risk;
  • connect trade-off to later evidence / dashboard.

Need:

Managers need defensible context, not just a record that something was late.

3. Manage Checkout Pull / Redeployment

Tasks:

  • detect queue pressure;
  • identify lowest-impact redeployment options;
  • record who was pulled, from where, for how long;
  • update affected department work;
  • release colleagues back when possible.

Need:

Service recovery should not invisibly break fill, fresh, dotcom or compliance.

Core Needs Covered

  1. Know real labour capacity
  2. Match work to skills and coverage
  3. Make checkout pull visible
  4. Route decisions to the right authority
  5. Preserve trade-off context

From: scattered execution evidence
To: role-specific control views

Management & Portfolio Visibility

State A

Store execution, evidence, blockers and actions are scattered across tasks, walks, WhatsApp, reports and manager memory.

State B

Deputy, Store Manager and Area Manager each see the right level of execution state, risk, evidence, root cause and follow-up action.

Steps 1-3 / create live visibility
Steps 4+ / manage portfolio follow-up

1. Build Deputy Live View

Tasks:

  • show active, blocked and expired work;
  • show coverage gaps;
  • show checkout pull;
  • show opening / closing readiness;
  • show what can be solved now vs escalated.

Need:

Deputy / Duty Manager needs to keep the shift stable in real time.

4. Build Area Manager Portfolio View

Tasks:

  • compare stores in a patch;
  • show which stores need attention first;
  • show process health, not only visual standards;
  • show action items and evidence;
  • show repeated blockers and root causes;
  • support escalation packs.

Need:

Area Manager needs portfolio-level control and credible root-cause context.

2. Build Store Manager Intervention View

Tasks:

  • show store readiness;
  • show top risks;
  • show labour vs demand;
  • show compliance reds;
  • show availability risk;
  • show open exceptions;
  • show AM actions due;
  • show carry-over.

Need:

Store Manager needs to know where to intervene, not inspect every task.

5. Close The Follow-Up Loop

Tasks:

  • track action items;
  • assign owner and due date;
  • show progress;
  • retain evidence;
  • carry unresolved items into the next review.

Need:

Management visibility only matters if agreed work actually closes.

3. Build Evidence Trail

Tasks:

  • link proof to task / exception;
  • preserve timestamps, actor, location and source;
  • separate strong, weak, missing and contested evidence;
  • support audit, compliance, HR, LP and AM needs.

Need:

Execution control and accountability require evidence, not recollection.

Core Needs Covered

  1. Give Deputy live shift control
  2. Give SM intervention visibility
  3. Preserve evidence and accountability
  4. Give AM portfolio-level process visibility
  5. Close action-item follow-up
